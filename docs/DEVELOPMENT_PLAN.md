# Zawer 研发规划

> 本文档为 Zawer 项目的完整研发规划，包含 6 个阶段、共 30+ 个 Todo 项。
> 每个 Todo 项包含：工作目的、修改内容、验收内容。

---

## 第一阶段：项目初始化与工程化搭建

> **阶段目标**：搭建前端工程化基础设施，确保团队开发规范统一，为后续功能开发提供稳定的开发环境。
>
> **预估周期**：1 周
>
> **核心交付物**：可运行的工程化项目骨架

### 1.1 初始化 Vite + React + TypeScript 项目

- **工作目的**：搭建前端项目骨架，确定技术栈基线
- **修改内容**：
  - 使用 `create vite` 创建 React + TypeScript 模板项目
  - 配置 `tsconfig.json` 的 `strict` 模式、路径别名（`@/` 指向 `src/`）
  - 配置 `vite.config.ts` 的 proxy 代理和构建选项
- **验收内容**：
  - [x] `npm run dev` 可正常启动开发服务器
  - [x] `npm run build` 可正常构建产物
  - [x] TypeScript 严格模式无报错
  - [x] 路径别名 `@/` 可正常使用

### 1.2 配置代码规范工具 ESLint + Prettier

- **工作目的**：统一代码风格，减少代码审查中的格式争议
- **修改内容**：
  - 安装并配置 ESLint（推荐 `@antfu/eslint-config` 或 `eslint-config-airbnb`）
  - 配置 Prettier 格式化规则（`.prettierrc`）
  - 配置 `.editorconfig`
  - 确保 ESLint 与 Prettier 规则不冲突
- **验收内容**：
  - [x] `npm run lint` 可正常执行且无报错
  - [x] `npm run format` 可自动格式化代码
  - [x] IDE 保存时自动格式化生效

### 1.3 配置 Git Hooks（Husky + lint-staged）

- **工作目的**：在代码提交前自动执行代码检查，防止不规范代码进入仓库
- **修改内容**：
  - 安装 Husky 和 lint-staged
  - 配置 `pre-commit` 钩子执行 lint-staged
  - 配置 `commit-msg` 钩子校验 Commit Message 格式（Conventional Commits）
- **验收内容**：
  - [x] `git commit` 时自动触发 ESLint 检查
  - [x] 不规范的 Commit Message 被拦截
  - [x] 仅对暂存区文件执行检查

### 1.4 集成 Tailwind CSS + ShadcnUI

- **工作目的**：搭建 UI 基础设施，提供统一的设计系统
- **修改内容**：
  - 安装并配置 Tailwind CSS v3+（`tailwind.config.ts`、`postcss.config.js`）
  - 初始化 ShadcnUI（`npx shadcn-ui@latest init`）
  - 配置主题色（Zawer 品牌色：红色系警告色 + 绿色系安全色）
  - 引入基础组件（Button、Card、Dialog 等）
- **验收内容**：
  - [x] Tailwind 工具类在组件中可正常使用
  - [x] ShadcnUI 组件可正常渲染
  - [x] 自定义主题色生效（红/绿色系）
  - [ ] 暗色模式切换正常（如适用）

### 1.5 搭建项目目录结构

- **工作目的**：建立清晰的模块化目录，便于团队协作和后续扩展
- **修改内容**：
  - 按功能模块划分目录：
    - `src/components/`（通用组件）
    - `src/features/`（功能模块：map、merchant、review、user）
    - `src/hooks/`（自定义 Hooks）
    - `src/services/`（API 请求层）
    - `src/stores/`（状态管理）
    - `src/types/`（TypeScript 类型定义）
    - `src/utils/`（工具函数）
    - `src/assets/`（静态资源）
    - `src/styles/`（全局样式）
- **验收内容**：
  - [x] 目录结构清晰，符合 Feature-based 架构
  - [x] 各目录下有 `index.ts` 统一导出
  - [x] README 中补充目录结构说明

### 1.6 配置状态管理与路由

- **工作目的**：为应用提供全局状态管理和页面路由能力
- **修改内容**：
  - 安装并配置 Zustand（轻量状态管理）
  - 安装并配置 React Router v6（路由管理）
  - 定义基础路由结构：首页（地图页）、商家详情页、搜索页、个人中心页
  - 创建 Layout 组件（顶部导航栏 + 底部 Tab 栏）
- **验收内容**：
  - [x] 页面间路由跳转正常
  - [x] 全局状态可跨组件共享
  - [x] Layout 布局在各页面中正确渲染
  - [x] 404 页面正常展示

### 1.7 配置 HTTP 请求层

- **工作目的**：封装统一的 API 请求工具，便于后续对接后端接口
- **修改内容**：
  - 安装 Axios
  - 封装请求/响应拦截器（统一错误处理、Token 注入、Loading 状态）
  - 定义 API 基础配置（baseURL、timeout）
  - 创建 Mock 数据服务（使用 MSW 或 json-server），模拟商家数据和评分数据
- **验收内容**：

  - [x] Mock 数据服务可正常返回模拟数据
  - [x] 错误拦截器可捕获并提示网络错误

### 1.8 配置环境变量管理

- **工作目的**：区分开发/测试/生产环境配置，保护敏感信息
- **修改内容**：
  - 创建 `.env.development`、`.env.production` 文件

  - [x] `.env` 文件不被提交到 Git
  - [x] `.env.example` 包含所有必要变量的说明

## 第二阶段：地图模块开发

> **阶段目标**：完成地图核心交互，实现用户定位、商家标注和 Zawer 等级可视化，这是产品的核心差异化功能。
>
> **预估周期**：1.5 周
>
> **核心交付物**：可交互的地图页面（含定位、标注、信息卡片）

### 2.1 集成高德地图 JS API 2.0

- **工作目的**：引入地图底层能力，为地图功能提供基础支撑
- **修改内容**：
  - 通过 `@amap/amap-jsapi-loader` 异步加载高德地图 SDK
  - 封装 `useAMap` Hook，管理地图实例的初始化和销毁
  - 创建 `MapContainer` 组件，作为地图的容器组件
  - 处理地图加载失败的降级方案（提示用户检查网络）
- **验收内容**：
  - [x] 地图可在页面中正常渲染
  - [x] 地图支持缩放、拖拽等基础交互
  - [x] 组件卸载时地图实例正确销毁，无内存泄漏
  - [x] 加载失败时展示友好提示

### 2.2 实现用户定位功能

- **工作目的**：获取用户当前位置，以用户为中心展示附近商家
- **修改内容**：
  - 使用高德地图 `AMap.Geolocation` 插件获取用户位置
  - 封装 `useGeolocation` Hook
  - 处理定位权限拒绝的场景（提示用户手动授权或输入位置）
  - 提供默认位置兜底（如成都市中心）
  - 在地图上展示用户当前位置标记（蓝色脉冲点）
- **验收内容**：
  - [x] 首次打开页面弹出定位授权请求
  - [x] 授权后地图自动定位到用户位置
  - [x] 拒绝授权时展示提示并提供手动输入入口
  - [x] 用户位置标记清晰可见

### 2.3 实现商家标注与 Zawer 等级可视化

- **工作目的**：在地图上直观展示商家位置和 Zawer 等级，实现"一目了然"的核心体验
- **修改内容**：
  - 根据 Mock 商家数据在地图上渲染自定义 Marker
  - Marker 颜色根据 Zawer 指数动态变化：5.0 深红、4.0 橙红、3.0 黄色、2.0 浅绿、1.0 深绿
  - Marker 上展示商家名称缩写和 Zawer 指数数字
  - 实现 Marker 聚合（`AMap.MarkerCluster`），避免密集区域标注重叠
- **验收内容**：
  - [x] 地图上可正确渲染商家标注
  - [x] 不同 Zawer 等级的标注颜色明显区分
  - [x] 缩放地图时标注聚合/展开正常
  - [x] 标注上的文字清晰可读

### 2.4 实现地图交互功能

- **工作目的**：提供流畅的地图交互体验，支持用户探索附近商家
- **修改内容**：
  - 点击 Marker 弹出商家简要信息卡片（InfoWindow）：商家名称、Zawer 指数、品类、距离
  - 卡片内提供"查看详情"按钮，跳转商家详情页
  - 实现地图范围变化时动态加载商家数据（`moveend` 事件）
  - 提供"回到我的位置"悬浮按钮
- **验收内容**：
  - [x] 点击标注弹出信息卡片，内容正确
  - [x] 点击"查看详情"可跳转到商家详情页
  - [x] 拖动地图后自动加载新区域商家
  - [x] "回到我的位置"按钮功能正常

### 2.5 地图性能优化

- **工作目的**：确保大量标注下地图仍流畅运行
- **修改内容**：
  - 实现可视区域内按需渲染（仅渲染视口内的 Marker）
  - 使用 `requestAnimationFrame` 优化批量渲染
  - 对商家数据请求做防抖处理（地图移动停止 300ms 后再请求）
  - 添加地图加载骨架屏
- **验收内容**：
  - [x] 500+ 标注下地图操作无明显卡顿
  - [x] 快速拖动地图不会触发大量重复请求
  - [x] 地图加载过程中展示骨架屏

## 第三阶段：核心业务功能开发

> **阶段目标**：实现商家详情、评分点评、搜索筛选等核心业务功能，构建完整的用户使用闭环。
>
> **预估周期**：2 周
>
> **核心交付物**：商家详情、评分点评、搜索筛选功能（前端 Mock 数据）

### 3.1 开发商家详情页

- **工作目的**：展示商家完整信息，帮助用户全面了解商家的 Zawer 状况
- **修改内容**：
  - 创建 `MerchantDetail` 页面组件
  - 页面包含：商家头图/轮播图、商家名称、Zawer 指数（大号醒目展示）、品类标签、地址、营业时间、联系电话
  - 展示 Zawer 指数雷达图（环境、服务、价格、质量等维度）
  - 底部展示用户点评列表
  - 提供"我要评分"悬浮按钮
- **验收内容**：
  - [ ] 商家详情页信息展示完整且布局美观
  - [ ] Zawer 指数以醒目的颜色和大小展示
  - [ ] 雷达图各维度数据正确渲染
  - [ ] 点评列表支持滚动加载
  - [ ] 页面在移动端和桌面端均正常展示

### 3.2 实现评分与点评功能

- **工作目的**：让用户可以对商家进行 Zawer 评分和撰写点评，构建众包数据体系
- **修改内容**：
  - 创建评分弹窗组件（`RatingDialog`）
  - 支持 1-5 分滑动评分（带表情反馈：😍→😡）
  - 支持多维度评分（环境、服务、价格、质量）
  - 点评表单：文字输入（限 500 字）+ 图片上传（最多 9 张）
  - 提交后实时更新商家 Zawer 指数
  - 展示评分分布柱状图
- **验收内容**：
  - [ ] 评分滑块操作流畅，表情随分数变化
  - [ ] 多维度评分可独立打分
  - [ ] 点评文字和图片可正常提交
  - [ ] 提交后商家 Zawer 指数实时更新
  - [ ] 空内容提交时有校验提示

### 3.3 开发点评列表与详情

- **工作目的**：展示其他用户的真实评价，帮助用户做出消费决策
- **修改内容**：
  - 创建 `ReviewList` 组件，展示点评列表（头像、昵称、评分、时间、内容、图片）
  - 支持按时间/评分排序
  - 支持点赞/举报功能
  - 点击图片可全屏预览
  - 实现分页加载（Infinite Scroll）
- **验收内容**：
  - [ ] 点评列表按时间倒序展示
  - [ ] 排序切换功能正常
  - [ ] 图片预览功能正常
  - [ ] 滚动到底部自动加载更多
  - [ ] 空列表展示"暂无点评"引导

### 3.4 实现搜索功能

- **工作目的**：让用户可以通过关键词快速找到目标商家
- **修改内容**：
  - 创建 `SearchPage` 页面
  - 顶部搜索栏支持关键词输入（带防抖 300ms）
  - 展示搜索历史（本地存储，最多 10 条）
  - 展示热门搜索推荐
  - 搜索结果以列表形式展示（商家名称、Zawer 指数、品类、距离）
  - 支持搜索结果在地图上高亮展示
- **验收内容**：
  - [ ] 输入关键词后可正确返回搜索结果
  - [ ] 搜索历史可保存和清除
  - [ ] 点击搜索结果可跳转商家详情
  - [ ] 无结果时展示友好提示

### 3.5 实现筛选功能

- **工作目的**：让用户可以按品类和 Zawer 等级快速过滤商家
- **修改内容**：
  - 在地图页和搜索页添加筛选栏
  - 品类筛选：餐饮、住宿、出行、购物、娱乐、其他（支持多选）
  - Zawer 等级筛选：极度 Zawer（4-5）、一般 Zawer（2.5-4）、不 Zawer（1-2.5）
  - 距离筛选：500m / 1km / 3km / 5km
  - 筛选条件变化时实时更新地图标注和列表
- **验收内容**：
  - [ ] 各筛选条件可正常选择和取消
  - [ ] 筛选结果实时反映在地图和列表中
  - [ ] 多条件组合筛选结果正确
  - [ ] 筛选条件可一键重置

### 4.3 实现收藏功能

## 第五阶段：后端 API 开发

> **阶段目标**：搭建后端服务，提供稳定的 API 接口，替换前端 Mock 数据，实现数据持久化。
>
> **预估周期**：2 周
>
> **核心交付物**：后端 API 服务 + 前后端联调完成

### 5.1 初始化后端项目（NestJS + PostgreSQL）

- **工作目的**：搭建后端工程基础设施
- **修改内容**：
  - 使用 NestJS CLI 创建项目
  - 配置 TypeORM 连接 PostgreSQL
  - 配置 Swagger API 文档
  - 配置 CORS、Helmet 等安全中间件
  - 配置环境变量管理（`@nestjs/config`）
- **验收内容**：
  - [ ] 后端服务可正常启动
  - [ ] 数据库连接成功
  - [ ] Swagger 文档页面可访问
  - [ ] API 支持跨域请求

### 5.2 设计并实现数据库模型

- **工作目的**：建立数据持久化基础
- **修改内容**：
  - 设计并创建数据表：

    - `merchants`（商家表）
    - `reviews`（点评表）
    - `ratings`（评分表）
    - `favorites`（收藏表）

  - 创建 Seed 数据（初始化测试商家数据）
- **验收内容**：
  - [ ] 所有数据表创建成功
  - [ ] 表间关联关系正确
